# Markdown 实时预览编辑器 — 演化实验总览

## 实验目的

用「变异 → 选择 → 保留」的演化回路，比较不同开发策略在**同一个任务**上产生的**代码差异**，
找出哪种策略更符合你的审美和工程价值观。

## 任务简介

实现一个 **Markdown → HTML 实时预览编辑器**，技术栈为 **React 18 + Tailwind CSS 3（CDN 引入，无需 Node.js）**。

## 三个变异版本

| 变体 | 策略 | 适合什么样的开发者 | 预判产物特征 |
|------|------|-------------------|-------------|
| **A: TDD 测试驱动** | 先写测试用例，再逐特性实现，测试全部通过才算完成 | 看重正确性，对边界情况敏感 | 解析器逻辑健壮，测试用例丰富，UI 可能较朴素 |
| **B: 架构先行** | 先做架构设计（数据流、组件树、接口），再按蓝图施工 | 看重可维护性和可扩展性 | 代码结构清晰，模块化好，文档完备，可能偏「过度设计」 |
| **C: MVP 渐进** | 30 分钟内出第一个可用的版本，然后持续迭代加功能 | 看重交付速度和快速反馈 | 第一个版本功能少但能用，后续版本逐渐丰满 |

## 使用方式

1. **阶段 1（变异）**：我已经生成了 3 个变体的 TASK.md，分别放在 `variant-a-tdd/`、`variant-b-arch/`、`variant-c-mvp/` 目录下
2. **阶段 2（执行）**：开 3 个 Claude Code（或其他 AI）窗口，每个窗口进入一个 variant 目录
   - 把对应 `TASK.md` 的内容作为任务 prompt 喂给 AI
   - 共享的 `SPEC.md` 可以一并交给 AI 作为需求文档
3. **阶段 3（选择）**：跑完后回到这个窗口，让我审查三个产物的差异
   - 我会从正确性、代码质量、可维护性、UI 体验等多个维度打分
   - 你给出人工判断：哪个最符合你的心意
4. **阶段 4（保留）**：我把你的选择写进 memory，下次生成变体时自动参考

## 目录结构

```
variants/
├── README.md                   ← 你正在看的文件
├── SPEC.md                     ← 共享的需求规格说明
├── variant-a-tdd/
│   └── TASK.md                 ← 变体 A 的任务书
├── variant-b-arch/
│   └── TASK.md                 ← 变体 B 的任务书
└── variant-c-mvp/
    └── TASK.md                 ← 变体 C 的任务书
```

## 评判维度（供参考，阶段 3 时会细化）

| 维度 | 权重 | 说明 |
|------|------|------|
| 功能完整度 | ★★★★★ | 支持的 Markdown 语法是否齐全？有没有遗漏？ |
| 边界处理 | ★★★★☆ | 嵌套语法、空输入、错误输入是否健壮？ |
| 代码可读性 | ★★★★☆ | 另一个程序员读你的代码，多久能理解？ |
| 架构设计 | ★★★☆☆ | 加一个新语法特性（如脚注）要改多少行？ |
| UI/UX 体验 | ★★★☆☆ | 界面好不好看？输入体验流畅吗？ |
| 性能 | ★★☆☆☆ | 大文本（10000 字）输入时是否卡顿？ |
